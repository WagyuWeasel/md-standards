# MD-Standards Universal Rules

These rules apply to every Markdown file generated by the md-standards system.
They are derived from CommonMark, GitHub Flavored Markdown, markdownlint,
Anthropic, Stripe, and Vercel documentation standards.


## Specification Compliance

1. **CommonMark first.** All files must comply with CommonMark spec v0.31.2.
   GFM extensions (tables, task lists, strikethrough, autolinks) are permitted
   where needed.

2. **markdownlint clean.** All files must pass `markdownlint` with zero
   warnings using the project's `.markdownlint.json` config.

3. **UTF-8 encoding.** All files must be UTF-8 with no BOM. Line endings: LF
   (Unix-style), never CRLF.


## Front Matter (YAML)

4. **Every file starts with YAML front matter** enclosed in `---` fences.
   Minimum required fields:

   ```yaml
   ---
   title: "Document Title"
   type: readme
   version: "1.0.0"
   created: 2026-02-28
   updated: 2026-02-28
   author: ""
   status: draft
   ---
   ```

5. **Optional fields** (include when relevant): `tags`, `audience`, `tldr`,
   `related`, `changelog`.


## Structure

6. **Single H1.** Every file has exactly one `# Title` heading, matching the
   front matter `title`.

7. **Heading hierarchy.** Never skip levels. H1 → H2 → H3 → H4. No orphan
   headings.

8. **Table of Contents.** Required for any file with 4+ H2 sections. Format:
   linked list under `## Table of Contents`. Auto-generate where possible.

9. **Blank lines.** One blank line before and after every heading, code block,
   list, blockquote, and table. Two blank lines before H2 sections (visual
   separation).

10. **Line length.** Soft wrap at 80 characters for prose. No hard limit for
    tables, URLs, or code blocks.

11. **Trailing whitespace.** None. No trailing spaces on any line.

12. **Final newline.** Every file ends with exactly one newline character.


## Content Quality

13. **Lead with purpose.** First paragraph after H1 states what this document
    is, who it's for, and why it exists. No preamble.

14. **Progressive disclosure.** Summary or TL;DR up top. Details below. Use
    collapsible `<details>` sections for advanced or optional content.

15. **One file, one purpose, one audience.** Never mix audiences or purposes
    in a single file.

16. **Active voice.** Default to active voice. Use imperative mood for
    instructions ("Run the command" not "The command should be run").

17. **Concise prose.** Write in clear, flowing paragraphs. Reserve bullet
    points for genuinely discrete items (3+ items that are parallel in
    structure). Bullet points must be at least one full sentence.

18. **No orphan links.** Every `[text](url)` must resolve. Descriptive link
    text always — never "click here" or "this link."


## Code

19. **Fenced code blocks.** Always use triple backticks with a language
    identifier: `` ```python ``, `` ```bash ``, `` ```yaml ``, etc. Never
    indented code blocks.

20. **Runnable examples.** Code examples should be copy-paste-runnable wherever
    possible. Include expected output in a separate block or comment.

21. **Inline code.** Use backticks for file names, commands, variables, keys,
    and paths: `README.md`, `npm install`, `API_KEY`.


## Media and Links

22. **Alt text on all images.** `![Descriptive alt text](image.png)` — never
    empty alt text.

23. **Relative links** for internal references within the same repo. Absolute
    URLs for external resources.

24. **Reference-style links** for documents with 5+ external URLs. Keeps prose
    readable.


## Accessibility and Portability

25. **No HTML unless necessary.** Use HTML only for features Markdown cannot
    express (e.g., `<details>`, `<kbd>`, anchor IDs). Never for styling.

26. **No platform-specific syntax.** Avoid features that only render on one
    platform (e.g., GitHub-only admonitions). Stick to CommonMark + GFM.

27. **Table readability.** Tables must be readable in raw text form. Align
    columns where practical. Use left-alignment by default.


## Naming and Organization

28. **File naming.** Lowercase, hyphens for spaces: `setup-guide.md`, not
    `Setup Guide.md` or `setup_guide.md`.

29. **One sentence per line** (optional but recommended for diff-friendly
    version control).
